# Feature Specification: SNAP Token Management Service & Endpoint

**Feature Branch**: `001-snap-token-management`

**Created**: 2026-07-22

**Status**: Draft

**Input**: User description: "buat codebase untuk project ini, lalu piloting dengan service dan endpoint manajemen token yang mengikuti aturan SNAP (Standarisasi Nasional Open API Pembayaran)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Request B2B Access Token via SNAP API (Priority: P1)

As an internal client system or external service consumer, I want to request a B2B OAuth access token using SNAP-compliant headers (`X-SIGNATURE`, `X-TIMESTAMP`, `X-CLIENT-KEY`) and payload format, so that I can securely authenticate subsequent API interactions with external payment systems and internal services.

**Why this priority**: B2B Token Management is the fundamental authentication entry point for SNAP (Standarisasi Nasional Open API Pembayaran). Without a valid SNAP-compliant token, no payment or bank gateway operations can occur.

**Independent Test**: Can be independently tested by executing a POST HTTP request to `/v1.0/access-token/b2b` with a valid RSA-SHA256 signature generated using the client's registered private key, and receiving a SNAP-standard 200 OK response containing `accessToken`, `tokenType`, and `expiresIn`.

**Acceptance Scenarios**:

1. **Given** a registered client with valid `X-CLIENT-KEY` and corresponding RSA public key on record, **When** the client submits a `POST /v1.0/access-token/b2b` request with valid `X-TIMESTAMP`, `X-CLIENT-KEY`, valid `X-SIGNATURE` (SHA256withRSA over `X-CLIENT-KEY|X-TIMESTAMP`), and body `{"grantType": "client_credentials"}`, **Then** the system MUST respond with HTTP 200 OK, `responseCode: "2007300"`, `responseMessage: "Successful"`, a valid signed JWT `accessToken`, `tokenType: "Bearer"`, and `expiresIn: "900"`.
2. **Given** an invalid or tampered `X-SIGNATURE` or mismatched `X-CLIENT-KEY`, **When** a client requests a token, **Then** the system MUST reject the request with HTTP 401 Unauthorized, `responseCode: "4017300"`, and `responseMessage: "Unauthorized. Invalid Signature"`.
3. **Given** a request missing required headers (`X-SIGNATURE`, `X-TIMESTAMP`, `X-CLIENT-KEY`) or invalid ISO 8601 timestamp, **When** a client requests a token, **Then** the system MUST reject the request with HTTP 400 Bad Request and standard SNAP error code (`4007300`).

---

### User Story 2 - Client Credential & Public Key Management (Priority: P2)

As a system administrator, I want to manage client credentials, client IDs, and public keys in the gateway persistent store, so that new partner systems or internal clients can be safely onboarded to generate SNAP tokens.

**Why this priority**: Enables secure onboarding and management of partner clients whose signatures are verified during token generation.

**Independent Test**: Can be tested independently by creating/updating a client record in the database with an RSA Public Key and verifying that the token service immediately validates signatures signed by the corresponding private key.

**Acceptance Scenarios**:

1. **Given** a new client onboarding request, **When** the administrator provisions a `client_id`, `client_secret`, and RSA Public Key in the storage layer, **Then** the system MUST store the client details securely and immediately accept signature verification for that client.
2. **Given** a revoked or inactive client account, **When** the client attempts to request a B2B token, **Then** the system MUST refuse token issuance with HTTP 401 Unauthorized and `responseCode: "4017300"`.

---

### User Story 3 - Access Token Validation & Cache Storage (Priority: P3)

As an internal payment gateway service, I want to validate incoming Bearer tokens against the active token cache/store, so that downstream payment services can verify client authorization rapidly with low latency.

**Why this priority**: Ensures fast, sub-millisecond token validation for high-throughput payment transaction endpoints.

**Independent Test**: Can be tested by generating a token, storing/caching it in Redis, and verifying that token validation passes for active tokens and fails for expired or revoked tokens.

**Acceptance Scenarios**:

1. **Given** a valid, active access token generated by the gateway, **When** validated by an internal service, **Then** the system MUST return token validity status, client context, and remaining TTL.
2. **Given** an expired token (> 900 seconds after issuance), **When** validation is attempted, **Then** the system MUST reject the token as invalid/expired.

---

### Edge Cases

- What happens if the `X-TIMESTAMP` in the request header is skewed beyond the allowed tolerance (e.g. ±5 minutes from current system time)? System MUST reject the request with `4007300 Invalid Field Format (Timestamp skew detected)`.
- What happens if a duplicate request with the same `Idempotency-Key` or signature replay is sent concurrently? System MUST enforce atomic locking in Redis and replay cached response without re-issuing duplicate active tokens.
- What happens if the primary database is temporarily unavailable during token verification? System MUST fall back to Redis cached client metadata or safely return standard 500 error code `5007300`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST expose a SNAP-compliant B2B token endpoint at `POST /v1.0/access-token/b2b`.
- **FR-002**: System MUST validate `X-CLIENT-KEY`, `X-TIMESTAMP` (ISO 8601 format within ±5 minutes tolerance), and `X-SIGNATURE` (SHA256withRSA signature over string component `client_id|X-TIMESTAMP`).
- **FR-003**: System MUST support `grantType: "client_credentials"` as specified in SNAP standards.
- **FR-004**: System MUST issue a signed JWT access token with a configurable default expiration time of 900 seconds (15 minutes).
- **FR-005**: System MUST return SNAP-standard JSON response structure:
  - `responseCode`: `"2007300"` for success, or SNAP error codes (e.g., `"4017300"`, `"4007300"`, `"5007300"`).
  - `responseMessage`: Human-readable status message.
  - `accessToken`: JWT Bearer string.
  - `tokenType`: `"Bearer"`.
  - `expiresIn`: Expiration time in seconds as string (e.g. `"900"`).
  - `additionalInfo`: Object for extensibility.
- **FR-006**: System MUST persist client credentials, client status, and RSA public keys in PostgreSQL and cache active client public keys in Redis.
- **FR-007**: System MUST store active tokens in Redis with expiration TTL matching `expiresIn` for fast validation and revoking capability.
- **FR-008**: System MUST emit OpenTelemetry traces, metrics, and structured Loki logs for every token request, tagging requests with `client_id`, `responseCode`, and `trace_id`.
- **FR-009**: System MUST enforce `Idempotency-Key` processing and Redis atomic locking to prevent race conditions on concurrent token generation calls.
- **FR-010**: System MUST run inside a Docker multi-stage build container under a non-root user (`USER appuser`).

### Key Entities

- **ClientApp**: Represents a registered partner or internal client system. Attributes include `id`, `client_id`, `client_name`, `status` (active/revoked), `created_at`, `updated_at`.
- **ClientKey**: Represents public key cryptographic material associated with a client. Attributes include `id`, `client_id`, `public_key_pem`, `key_id`, `algorithm` (e.g., RSA-256), `is_active`.
- **AccessTokenSession**: Represents an active issued token session. Attributes include `token_id`, `client_id`, `jti`, `issued_at`, `expires_at`, `is_revoked`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% compliance with Bank Indonesia SNAP B2B Access Token API contract specifications.
- **SC-002**: Token generation endpoint `POST /v1.0/access-token/b2b` p95 response latency is under 50ms under normal operating load.
- **SC-003**: 100% of invalid signatures, expired timestamps, or revoked client credentials are correctly rejected with appropriate SNAP HTTP status and error codes.
- **SC-004**: 100% of token requests produce trace context in Grafana Tempo and correlated structured logs in Grafana Loki.

## Assumptions

- Client systems hold their RSA private keys securely and provide their RSA public key during onboarding in PEM format.
- Clock drift across client servers and gateway servers is within ±5 minutes (enforced via `X-TIMESTAMP` validation).
- PostgreSQL migration scripts manage table creation for `client_apps` and `client_keys`.
